## 一、這個工具現在是什麼

一套**純前端、免安裝、可離線**的 TOEIC 900+ 衝刺練習軟體，涵蓋 TOEIC 除了 Part 1（看圖）以外的所有題型：

| 模組 | 題量 | 用途 |
|---|---|---|
| 單字卡（SRS） | 621 個 900+ 進階詞彙 | 間隔重複記憶，自動發音 |
| 聽力 Part 2 | 60 題 | 應答問題，專攻間接回應與同字陷阱 |
| 聽力 Part 3/4 | 13 組（39 題） | 對話／短講，男女聲區分，先讀題再聽 |
| Part 5 文法 | 200 題 | 單句文法，類別弱點統計 |
| Part 6 段落填空 | 10 篇（40 格） | 含句子插入題 |
| Part 7 閱讀 | 12 組（44 題） | 單篇→雙篇→三篇，限時訓練 |
| **合計** | **逾 1,000 題** | 可再用「一鍵擴增」持續增加 |

外加：**完整模擬考**（仿真兩節測驗）、**錯題本**（答錯自動收錄、答對才移除）、**統計與估分**、**雲端同步**（手機↔電腦進度一致）、**備份匯出／還原**、**三階段週期化計畫引擎**（依考試日期自動切換每日菜單）。

**品質狀態**：2026-07-12 通過一次獨立的正式驗收稽核——1,004 題抽樣人工複審零錯誤、核心功能逐一動態驗證通過、修掉一個 XSS 安全漏洞與兩個無障礙缺陷。這是一個功能完整、品質扎實的**個人專案／小型產品等級**成品；離「大型商業補習平台」的唯一實質差距是**內容數量**（見第四節）。

## 二、推薦使用方式（你的情境：600–750 → 900+，考期約 2026/8 底）

### 每天的鐵律

1. **先在首頁設定考試日期**——這是整個工具的引擎開關，三階段（基礎→強化→衝刺）會依日期自動切換，每日菜單也跟著變。
2. **照首頁「今日菜單」逐項做**，全部打勾＝今天完成。菜單是依你所處階段自動排的，不用自己想順序。
3. **每天一定要出席**——SRS 單字與錯題本都是「拖越久越虧」的機制，斷一天的成本比想像大。連續天數是這個計畫成敗的第一因素。

### 依你目前的時間點（約考前 6–7 週，正處基礎期尾聲）

- 這 1–2 週還在**基礎期**：重點是「文法零漏洞＋Part 2 反應速度＋單字第一輪過完」。Part 5 每個類別的正確率先拉到 85% 以上，是後面省時間的本錢。
- 大約 **7 月下旬進入強化期**：主戰場換成聽力 Part 3/4 與多篇閱讀。**進入強化期後養成一個關鍵習慣——聽力音檔播放前，先用幾秒把三個題目掃過一遍**，這一個動作對 Part 3/4 的提升最明顯。
- **8 月中進入衝刺期**：不再學新內容，只做限時訓練＋錯題本每天清零＋保持手感。

### 各模組的用法要點

- **單字卡**：翻面後**誠實評分**是整個 SRS 的靈魂。「模糊」硬按「記得」只是把遺忘延後到考場上。按鈕上的天數就是下次出現的間隔，間隔 ≥21 天代表精通。
- **聽力**：答錯先按「↻ 重播」聽懂，再看逐字稿——**不要一答錯就直接看文字**，那樣練不到耳朵。每組聽完後跟著逐字稿再聽一次（shadowing 跟讀）效果加倍。
- **Part 5/6**：每題逼自己 20 秒內作答，看完詳解再下一題。曾答錯與沒做過的題會**優先出**，所以持續練會自動聚焦在你的弱項。
- **Part 7**：計時器超時會變紅但不會中斷——目的是訓練你「感覺到時間」，而不是逼你猜題。NOT 題與跨篇整合題是 900 分的關鍵題型。
- **模擬考**：戴耳機、確認有 22 分鐘不被打擾再開始。它是**迷你模考**（約 44–46 題、35 分鐘），用來練節奏與抓弱項；交卷後的「弱項 Top 3」點下去直接跳到對應練習。**900+ 仍需要每週末補一回官方等級的完整 2 小時模擬題**（推薦 ETS 官方全真試題指南），補足題型廣度與體力訓練。

### 每週日的 15 分鐘檢討（看統計頁）

1. 預估分數有沒有上升？（僅供追蹤趨勢，實測通常略低，別當真實分數）
2. Part 5 哪個類別最弱？下週會自動優先出錯過的題。
3. 聽力 Part 2 vs Part 3/4 哪個低？低的那個下週多練一輪。
4. 連續學習天數斷了嗎？

## 三、資料安全與備份（重要，別跳過）

- 學習紀錄**即時自動存**在你這台瀏覽器的 localStorage。**固定用同一台電腦的同一個瀏覽器**，且**別清除瀏覽資料**，否則紀錄會消失。
- 想讓手機與電腦看到同一套進度 → 設定**雲端同步**（一次性約 3 分鐘，步驟見 README）。設定後兩台裝置輸入同一組同步代碼即可。
- **換裝置、換瀏覽器、或要清瀏覽資料之前，先到「題庫/備份」分頁匯出一份備份檔**。超過 7 天沒備份，首頁會提醒。
- 備份檔只含學習紀錄＋自訂題庫，**不含同步代碼**，所以分享備份檔不會外洩你的雲端讀寫權（這點在稽核中已驗證）。
- 同步代碼的安全等級是「知道代碼的人才能讀寫」，足夠個人讀書進度用，**不要拿來存機敏資料**。

## 四、未來升級藍圖

依「投資報酬率」排序，分三個層級。**第一層是我建議優先做的。**

### 🥇 第一層：高價值、低成本（建議近期做）

1. **iOS Safari 聽力自動播放修正**（唯一的 Blocker 風險項）
   - 問題：iOS 規定「使用者手勢後才能發聲」，而目前聽力是「進頁面就自動播放」，在 iPhone Safari 上很可能一進去是**無聲**的。這是聽力＝本產品核心功能，卻是自動化測不出來的真機專屬問題。
   - 建議解法：聽力練習首次播放改成需要**點一下播放鈕**觸發，之後同一組內才自動連播。改動不大，但直接決定 iPhone 使用者能不能用核心功能。
   - **請先在 iPhone 上實測確認是否真的無聲**（已列入 README 真機清單第一項），確認後我可以動手修。

2. **真正的離線 PWA（Service Worker）**
   - 現況：已有 manifest、可「加入主畫面」，但沒有 Service Worker，飛航模式或沒網路時第一次載入會失敗。
   - 升級：加一個 Service Worker 快取所有靜態資源，讓工具在完全沒網路時也能開（對通勤／飛機上背單字很有用）。成本低、體驗提升明顯。

3. **估分模型校準**
   - 現況：預估分數用的是簡單線性換算曲線，只能看趨勢。
   - 升級：等你累積幾次官方模考或正式成績後，用真實分數回歸校準換算曲線，讓「預估分數」更貼近實測，週檢討時更有參考價值。

### 🥈 第二層：高價值、高成本（決定產品天花板）

4. **內容深度擴充：1,000 題 → 3,000–5,000 題**
   - 這是離「大型商業平台」唯一的實質差距，也是最耗人力的一項。
   - 好消息：工具已內建「一鍵擴增」——AI 產生指令 → 貼回 JSON 自動驗證去重合併。可以**分批、持續**擴充，不必一次到位。建議優先擴充**聽力 Part 3/4** 與 **Part 7 多篇閱讀**（這兩個是 900 分的關鍵，也是目前相對最少的）。
   - 每次擴充後記得跑一次內容抽審（沿用稽核的做法），確保新題的正解唯一且詳解不引用選項字母。

5. **真人錄音音檔選項**
   - 現況：聽力用瀏覽器語音合成（TTS），免音檔、免流量，但語調不如真人自然。
   - 升級：對正式模考題提供真人錄音選項（例如錄一批高品質音檔放 CDN）。好處是更接近真實考場聽感，代價是內容製作成本與儲存流量。可作為「進階付費內容」的雛形。

### 🥉 第三層：錦上添花（有餘力再考慮）

6. **Part 1 看圖題**：TOEIC 有但本工具沒有，因為需要圖片素材。若要補齊題型完整度可加，但需要準備配圖。
7. **更深入的學習分析**：例如「同一文法類別連續錯 3 次自動推播加強」、單字遺忘曲線視覺化、每日/每週學習時間統計。
8. **社群／競爭功能**：好友排行榜、讀書會共享進度。這會需要真正的後端與帳號系統，複雜度跳一個量級，除非要往多人平台走，否則不急。
9. **無障礙深化**：目前已過對比度與基本 ARIA，可再做一次完整螢幕閱讀器逐頁走查。

### 一句話總結升級順序

> **先修 iOS 聽力（不然 iPhone 使用者用不了核心功能）→ 加離線 PWA →（長線）持續用一鍵擴增把聽力與閱讀題庫做厚**。前兩項是「補齊體驗漏洞」，第三項是「決定產品能走多遠」。

## 五、常見問題

**Q：聽力沒聲音？**
A：需要用 Edge 或 Chrome（使用系統內建語音合成）。iPhone Safari 目前可能因自動播放限制而無聲（見升級藍圖第 1 項）。

**Q：換了新電腦，紀錄不見了？**
A：localStorage 綁定單一瀏覽器。換裝置前要先匯出備份檔還原，或設定雲端同步。

**Q：預估分數準嗎？**
A：只能看趨勢。實測含時間壓力與體力因素，通常比工具預估低 20–50 分，這是正常的。

**Q：可以自己加題目嗎？**
A：可以。「題庫/備份」分頁有「一鍵擴增」，用 AI 產生指令、貼回 JSON 即可。詳解記得**引用選項內容、不要寫 (A)(B)(C)**，因為選項顯示時會洗牌。

## 六、技術架構附錄（維護者用）

- **技術棧**：純 vanilla JS + HTML + CSS，無框架、無建置流程。localStorage 存資料，Firebase Realtime Database 做雲端同步。
- **部署**：GitHub Pages（repo [realsnova/toeic-900](https://github.com/realsnova/toeic-900) master 分支）。改完程式要 `git push`，且**務必先遞增 index.html 的 `?v=N` 版本號**避免瀏覽器快取舊檔。
- **檔案結構**：`js/app.js`（主邏輯）、`js/mock.js`（模擬考）、`js/ui.js`（Settings/Theme/Accents/UI/Anim 模組）、`js/views.js`（hub 頁 renderer）、`js/sync.js`（雲端同步）、`js/data/*.js`（題庫）。
- **兩條不可踩的線**：
  1. **開機流程與 `renderers.home()` 路徑上的任何程式碼都不可呼叫 `save()`**——會搶在雲端同步比對前污染時間戳，蓋掉真實進度。
  2. **任何要當 Firebase key 的字串（`S.cat[x]`、`S.vocab[x]` 等）不可含 `. # $ / [ ]`**——會讓雲端同步整批靜默失敗。使用者匯入路徑已加清洗，新增功能時要記得這條。
- 詳細的開發教訓與環境限制記錄在專案 memory 檔。
